AI Portfolio Analyzer (AIPort)

Complete Project Explanation & Interview Guide

MERN Stack • Groq AI (Llama 3.3 70B) • GitHub API • JWT Authentication

🎯 One-Line Pitch

"AIPort is a full-stack MERN application that uses AI (Groq/Llama 3.3) to analyze a developer's GitHub profile, portfolio, and resume — generating personalized scores, metrics, and actionable career suggestions."

📌 Problem Statement

Developers often struggle to understand how their online presence (GitHub, portfolio, resume) is perceived by recruiters. There's no easy way to get an objective, data-driven assessment of their tech profile. AIPort solves this by aggregating real GitHub data and feeding it to an LLM to produce a structured, personalized analysis report.

🔄 Complete Working Flow

Step 1: User Registration

User fills form → POST /api/auth/register → bcrypt hashes password →
Save to MongoDB Users collection → Generate JWT → Return token to frontend →
Store in localStorage via AuthContext

Step 2: User Login

User enters credentials → POST /api/auth/login → Find user by email →
bcrypt.compare(password) → Generate JWT → Return token → Store in localStorage

Step 3: Submit Analysis (Core Feature)

User enters GitHub username + Portfolio URL + optional Resume PDF
    ↓
Frontend sends FormData with Bearer token → POST /api/analysis
    ↓
protect middleware → Extracts JWT from Authorization header → jwt.verify() → Attaches req.user
    ↓
analysisController.submitAnalysis() runs

3a. GitHub Data Collection (3 API calls)

1. GET /users/{username}         → Profile: bio, followers, repos count, account age
2. GET /users/{username}/repos  → Top 30 repos: languages, stars, forks, topics
3. GET /users/{username}/events → Recent activity: pushes, PRs, issues

Aggregated data includes:

  • Total stars & forks across all repos
  • Language breakdown (e.g., "JavaScript (12 repos), Python (5 repos)")
  • Top 8 repos ranked by stars with descriptions
  • Recent activity counts (push events, PRs, issues)
  • Profile completeness (bio, blog, hireable flag, profile README)

3b. AI Analysis (Groq API Call)

Build rich prompt with ALL GitHub data →
POST to Groq API (Llama 3.3 70B model) →
LLM returns structured JSON: { scores, metrics, suggestionItems, jobReadiness }

3c. Save & Return

Parse AI JSON response → Create MongoDB Analysis document →
Link to user via userId → Return to frontend → Navigate to /analysis

Step 4: View Analysis Report

GET /api/analysis → Fetch user's analyses sorted by date →
Display: Radar chart, Score cards, Progress bars, Suggestions, Job readiness badge

Step 5: Admin Dashboard

Admin login → protect middleware → adminProtect (role === 'admin') →
GET /api/admin/users → List all registered users
GET /api/admin/analytics → Total users, total analyses, recent reports

Key Security Decisions:

  • Passwords are hashed with bcrypt (10 salt rounds) — never stored in plaintext
  • JWT tokens expire after 30 days
  • The protect middleware extracts the token from the Authorization: Bearer <token> header
  • Admin routes have a two-layer middleware: first protect (authenticate), then adminProtect (authorize role)

🤖 AI Analysis Pipeline — Deep Dive

Why Groq + Llama 3.3?

  • Groq provides the fastest LLM inference (hundreds of tokens/sec) — the analysis feels near-instant
  • Llama 3.3 70B Versatile is open-source, highly capable, and free on Groq's API tier
  • temperature: 0.3 balances consistency with slight variation between analyses

How I Prevent Generic/Hallucinated Results

  1. Real data injection: GitHub API data is fetched BEFORE the AI call — the prompt contains verified numbers (stars, followers, repo names)
  2. Metrics are hardcoded: totalProjects, followers, stars come directly from GitHub API, NOT from the AI
  3. Specific prompt engineering: The prompt explicitly says "Each suggestion MUST reference the developer's SPECIFIC repos, languages, or gaps by name"
  4. Structured JSON output: The AI is forced to return a strict JSON schema — any deviation is caught by JSON.parse()

Error Handling

  • If GitHub API fails → graceful fallback with error message in prompt
  • If Groq returns markdown-wrapped JSON → regex strips ```json fencing
  • If JSON parsing fails → error is thrown and returned as 500 to user

🎤 Interview Q&A — Anticipated Questions

Q1: "Why MERN stack?"

MongoDB's schema flexibility is ideal because AI responses can vary in structure. Express + Node gives me non-blocking I/O for handling multiple GitHub API calls concurrently. React's component architecture made the dashboard modular and reusable.

Q2: "Why not use OpenAI? Why Groq?"

Groq provides the fastest LLM inference available — ~500 tokens/sec vs OpenAI's ~80. For a real-time analysis tool, speed is critical so users don't wait 30+ seconds. Llama 3.3 70B is open-source and performs comparably to GPT-4 for structured tasks. Groq also has a generous free tier.

Q3: "How do you prevent the AI from hallucinating scores?"

The key metrics (totalProjects, followers, stars) are fetched directly from GitHub's API and hardcoded into the JSON template — the AI doesn't generate these numbers. For scores, the prompt includes detailed scoring rubrics tied to real data points, and I set a low temperature (0.3) for consistency.

Q4: "How does authentication work?"

I use stateless JWT authentication. On login, the server verifies the password using bcrypt, generates a JWT signed with a secret key, and sends it to the client. The client stores it in localStorage and includes it as a Bearer token in the Authorization header for every protected API call. The protect middleware extracts and verifies this token on each request.

Q5: "What happens if the GitHub API rate-limits you?"

Unauthenticated GitHub API calls allow 60 requests/hour per IP. For production, I'd add a GitHub Personal Access Token (5,000/hour). The code handles failures gracefully — if the GitHub fetch fails, the analysis continues with limited data and the AI acknowledges the unavailable data.

Q6: "Why store analysis results in MongoDB instead of just returning them?"

Storing enables users to view their analysis history, track progress over time, and the admin dashboard to aggregate platform-wide analytics. It also avoids redundant API calls if the user refreshes the page.

Q7: "How is the admin panel secured?"

Two-layer middleware: first protect verifies the JWT token (authentication), then adminProtect checks if req.user.role === 'admin' (authorization). The frontend also hides admin routes using conditional rendering, but the real security is server-side.

Q8: "What's the most challenging part you built?"

The AI analysis pipeline. Getting the LLM to return consistently parseable JSON while being specific to each user required careful prompt engineering. The first version gave generic suggestions to everyone because I was only passing 2 data points. I solved it by fetching detailed repo data, language breakdowns, and activity patterns, then instructing the AI to reference specific repos by name.

Q9: "How would you scale this for production?"

• Add Redis caching for GitHub API responses (cache for 1 hour)
• Move JWT token to httpOnly cookies to prevent XSS
• Add rate limiting on the analysis endpoint (express-rate-limit)
• Deploy backend on AWS/Railway, frontend on Vercel
• Use MongoDB Atlas for managed database
• Add GitHub OAuth login flow instead of username input
• Queue analysis jobs with Bull/BullMQ for heavy traffic

Q10: "What's the FormData approach for file upload?"

The frontend creates a FormData object and appends all fields. Axios automatically sets Content-Type to multipart/form-data with the correct boundary. On the backend, Multer middleware parses the request, saves the file to uploads/, and exposes req.file and req.body.

Q11: "Explain the Context API usage."

I used React's Context API instead of Redux because the app has simple global state — just the user object. The AuthProvider wraps the entire app, providing user, login(), register(), logout(), and loading via useContext(AuthContext). User data persists in localStorage across page refreshes.

Q12: "How do protected routes work on the frontend?"

React Router v7's <Navigate> component acts as a guard. /dashboard checks if user exists — if not, redirects to /login. /admin additionally checks user.role === 'admin'. This is UX-level protection; the real security is the backend middleware.

Q13: "What design patterns did you use?"

• MVC Pattern: Models (Mongoose), Views (React), Controllers (Express handlers)
• Middleware Chain: protect → adminProtect → controller for layered security
• Context Provider Pattern: Centralized auth state management
• Separation of Concerns: GitHub fetching is a separate function from AI analysis logic

Q14: "How does dark mode work?"

The ThemeToggle component toggles a dark class on the <html> element. Tailwind's dark: prefix applies alternate styles. The theme is stored in localStorage, so it persists across sessions.

Q15: "If you had more time, what would you add?"

• GitHub OAuth: Connect GitHub directly instead of typing username
• Resume PDF parsing: Use pdf-parse to extract text for the AI prompt
• Analysis comparison: Side-by-side comparison of reports over time
• Scheduled re-analysis: Auto re-analyze profiles weekly and email results
• Portfolio scraping: Actually fetch and analyze portfolio website content
